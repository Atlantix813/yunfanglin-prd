## 需求背景

### 1. 张教授侧需要一套轻量教培工具

2026-05-15 会议中，张教授认可先做一版可试用的教培系统。当前核心诉求聚焦课程上传、课程展示、学员观看和学习统计，在线录制、剪辑和直播暂缓。

本期先覆盖最基础的教学闭环：老师或运营人员线下录好视频，后台上传视频并配置课程，居民或工作人员在移动端观看课程，后台查看谁学了、学到什么程度。

### 2. 原 MeEdu 能力可复用，但原产品偏知识付费

MeEdu 已经具备资源库、录播课、章节、课时、视频播放、学员学习记录、后台管理等能力，适合作为教培系统底座复用。

但 MeEdu 原始产品包含课程售卖、VIP 会员、订单、优惠码、交易统计、商品数据、装修、评论等知识付费和运营能力。本期云芳邻教培不做课程售卖，这些内容如果保留，会让演示和后续研发理解偏向付费网校，增加误操作和评审风险。

### 3. 云芳邻需要承接账号和组织归属

教培系统接入后沿用云芳邻用户身份。云芳邻已有用户、手机号、组织归属、区县、街道、社区、用户角色等基础资料，教培系统应从云芳邻入口进入，并在进入时自动生成教培系统内部学员或管理员映射。

这样后台统计才能按街道、社区查看学习情况，也能避免用户再次注册、再次登录、重复维护头像昵称和组织资料。

### 4. 本期先服务 6 月底试用目标

张教授希望 6 月底前看到可试用版本。为了保证交付可控，本期优先保留“能上传课程、能学习、能统计”的主线能力，暂缓在线录制、视频剪辑、直播、付费、会员和复杂运营能力。

| 交付效率 | 尽量复用 MeEdu 已稳定的课程和学习能力，只做云芳邻接入、展示清理和必要统计改造。 | 开发能按页面逐项确认“保留什么、去掉什么、改造什么”。 | 全部 PC 端和移动端原型说明 |

| 判断项 | 通过后 | 不通过时 |
| --- | --- | --- |
| 用户已注册 | 继续判断个人信息 | 先完成云芳邻注册 |
| 个人信息已完善 | 继续判断可学习角色 | 先进入云芳邻资料完善页面 |
| 拥有可学习角色 | 进入移动端教培系统 | 不展示入口，直接访问也要拦截 |

#### ② 首次进入怎么处理

1. 系统用云芳邻用户 ID 查找教培系统学员 ID。

4. 已有学员映射时，刷新基础资料，学习记录继续保留。

- 未完善个人信息的用户不能直接进入移动端教培系统。

- 首次进入后，后台【学员列表】能看到对应学员。

## PC端 · 教培系统入口

### 1. 定位 · 教培系统入口是账号接入总入口

#### ① 这个说明做什么

【教培系统入口】不单独描述某一个业务页面。它用于说明云芳邻如何进入教培系统、哪些用户能进、进入时如何生成教培系统账号映射。

本期按“云芳邻用户是唯一外部身份”处理。

移动端教培系统要求用户已注册、已完善个人信息，并符合可学习角色。

PC 后台要求后台用户已登录，并拥有教培后台管理角色。

角色名称暂不定死，先预留为「可学习角色」和「教培管理员」两类。

同一个云芳邻用户进入移动端教培系统时生成学员映射，进入 PC 后台管理时生成管理员映射。教培系统内部继续保留学员 ID 和管理员 ID，但这些 ID 只作为内部记录使用，不作为用户登录入口。

#### ② 入口和账号关系

| 场景 | 用户从哪里进入 | 进入条件 | 首次进入后生成什么 | 用途 |
| --- | --- | --- | --- | --- |
| 移动端学习 | 云芳邻移动端学习入口 | 已注册、已完善个人信息、拥有可学习角色（角色待定） | 教培系统学员 ID | 课程学习、课时观看、学习记录 |
| PC 后台管理 | 云芳邻 PC 后台「教培系统」菜单 | 后台用户已登录，拥有「教培管理员」或后续确定的管理角色 | 教培系统管理员 ID | 资源管理、课程管理、学习记录查看 |

#### ③ 角色怎么配置

| 角色/身份 | 在哪里配置 | 用户会看到什么 | 说明 |
| --- | --- | --- | --- |
| 可学习角色（待定） | 云芳邻用户资料或角色配置 | 移动端教培系统入口 | 只有符合角色要求且资料完善的用户才能进入学习 |
| 教培管理员（待定） | 云芳邻后台用户权限 | PC 后台出现「教培系统」菜单 | 进入后免登录打开教培后台 |

### 2. PC 后台进入教培系统

#### ① 用户怎么进入

后台人员登录云芳邻 PC 后台后，系统判断这个用户是否拥有教培后台管理角色。满足时，左侧显示「教培系统」一级菜单；点击后，右侧直接打开教培系统后台，不再出现教培系统自己的登录页。

#### ② 进入流程

1. 用户登录云芳邻 PC 后台。
2. 系统检查用户是否拥有教培后台管理角色。
3. 校验通过后，云芳邻 PC 后台显示「教培系统」菜单；校验不通过时不显示入口。
4. 用户点击「教培系统」后，系统用云芳邻用户 ID 查找教培系统管理员 ID。
5. 如果还没有管理员映射，系统自动生成教培系统管理员 ID，并记录首次进入时间。
6. 如果已有管理员映射，系统更新最近进入时间，并直接进入教培后台。
7. 如果用户后续失去教培后台管理角色，云芳邻 PC 后台不再显示「教培系统」菜单，直接访问也要拦截。

#### ③ 需要记录哪些信息

| 信息 | 来源 | 用途 |
| --- | --- | --- |
| 云芳邻用户 ID | 云芳邻账号 | 识别真实人员 |
| 教培系统管理员 ID | 首次进入时生成 | 对应教培后台操作记录 |
| 首次进入时间 | 教培系统记录 | 判断映射创建时间 |
| 最近进入时间 | 教培系统记录 | 判断后台使用情况 |

### 3. 移动端进入教培系统

#### ① 用户怎么进入

居民或工作人员从云芳邻移动端进入学习中心后，系统先判断用户是否已注册、个人信息是否已完善、是否拥有可学习角色。全部满足时，直接进入移动端教培系统，不展示教培系统自己的登录入口。

#### ② 进入流程

1. 用户从云芳邻移动端点击学习入口。
2. 系统检查用户是否已注册；未注册时先完成注册。
3. 系统检查个人信息是否已完善；未完善时先进入资料完善页面。
4. 系统检查用户是否拥有可学习角色；不符合时不展示入口，直接访问也要拦截。
5. 校验通过后，系统用云芳邻用户 ID 查找教培系统学员 ID。
6. 如果还没有学员映射，系统自动生成教培系统学员 ID，并写入云芳邻用户基础资料和组织归属。
7. 如果已有学员映射，系统刷新昵称、手机号、区县、街道、社区、用户角色等基础资料，学习记录继续保留。

#### ③ 学员资料以谁为准

学员资料以云芳邻为准。昵称、手机号、区县、街道、社区、用户角色等信息发生变化后，进入教培系统时应刷新到教培系统展示层；课程学习记录、课时观看记录继续保留在教培系统内。

### 4. 验收重点

- 没有「教培管理员」权限的后台用户看不到「教培系统」菜单。
- 有权限的后台用户点击「教培系统」后，不出现二次登录。
- 个人信息未完善的用户不能直接进入移动端教培系统，应先完成资料。
- 同一个云芳邻用户进入移动端教培系统时生成教培系统学员 ID，进入 PC 后台时生成教培系统管理员 ID。
- 云芳邻取消「教培管理员」权限后，该用户不能再进入教培后台。
- 教培后台操作记录能追溯到云芳邻用户 ID 和教培系统管理员 ID。

## PC端 · 资源管理-图片库

### 1. 保留 · 图片库作为已上传图片管理页

#### ① 这个页面做什么

【图片库】用来查看、筛选和删除教培系统里已经上传过的图片。

运营人员不需要为了创建课程提前来这里上传图片。更常见的使用方式是：在新建或编辑录播课时直接上传封面，系统自动把这张图收进【图片库】，并归到「课程封面」分类。

#### ② 图片分类怎么用

图片库左侧的分类是“从哪里上传来的”。

| 分类 | 用户怎么理解 | 常见来源 |
| --- | --- | --- |
| [全部图片] | 查看所有图片 | 不限制来源 |

| [其它] | 未明确归类的图片 | 兜底分类 |

点击左侧分类后，右侧只展示对应来源的图片。点击「上传图片」时，上传的新图片会归到当前选中的分类。点击「删除」前必须先选中图片。

### 2. 调整 · 图片库接入云芳邻后的处理要求

#### ① 保留现有能力

本期 MVP 保留原系统现有的图片上传、查看、来源筛选、分页和预览能力。【图片库】仍然使用教培系统原有图片接口和存储配置，保证课程封面、图文介绍图片能继续正常读取。

#### ② 需要处理的点

| 事项 | 处理要求 | 原因 |
| --- | --- | --- |
| [删除] 操作 | MVP 阶段保留原能力；正式上线前至少增加二次确认 | 当前是嵌入使用，不建议为资源库单独做重改 |
| [图片使用情况] | 暂不作为 MVP 必做；后续如要二开，可增加“被课程封面使用中”等提示 | 现有资源库没有直接记录图片被哪里引用，课程封面保存的是图片地址，需要额外反查 |

#### ③ 验收重点

- 在【录播课管理】上传课程封面后，【图片库】的「课程封面」分类能看到该图片。
- 删除图片前有明确确认提示，避免运营人员误删素材。

## PC端 · 资源管理-视频库

<!-- prototype: src="prototype/pc-resource-video-source.html" width="1920" height="1080" scale="0.55" title="PC端原型 · 资源管理-视频库" -->

### 1. 保留 · 视频库作为课时视频管理页

#### ① 这个页面做什么

【视频库】用来管理已经上传到教培系统的视频素材。它服务的是课时创建和课时维护，不直接面向学员展示。

运营人员可以在课时管理中上传或选择视频。视频上传完成后，会进入【视频库】，后续可以按分类和关键字查找，再绑定到具体课时。

#### ② 页面保留哪些能力

| 区域 | 保留能力 | 说明 |
| --- | --- | --- |
| [分类树] | 按视频分类筛选 | 用于把培训视频按主题、用途或阶段分开管理 |
| [操作栏] | 「添加分类」「上传视频」「修改视频分类」「删除视频」 | 沿用原系统现有后台操作 |
| [搜索区] | 按关键字筛选视频 | 用于快速查找已上传视频 |
| [表格] | 展示视频名称、时长、大小、加密状态、上传时间 | 让运营人员确认素材是否可用于课时 |
| [分页] | 按页查看视频素材 | 沿用原系统现有分页能力 |

#### ③ 添加分类怎么用

点击「添加分类」后，系统打开原系统现有分类弹窗。运营人员需要选择「所属上级」，再填写「分类名称」。

「所属上级」可以选择「作为一级分类」，也可以选择已有分类作为上级。分类添加成功后，会出现在左侧分类树里，用于后续上传视频、筛选视频和调整视频归属。

这个分类只用于后台管理视频素材，不等同于学员端看到的课程分类。学员端课程展示仍然跟随课程管理里的课程分类。

#### ④ 上传视频怎么用

点击「上传视频」后，系统打开原系统现有上传弹窗。运营人员选择视频并上传成功后，视频进入当前分类，后续可在课时管理中绑定。

如果视频存储配置未完成，原系统源码会提示「请先在系统配置的视频存储中完成参数配置」。本期嵌入时要保证视频存储配置可用，否则课程可以创建，但课时无法正常承载视频学习。

### 2. 调整 · 视频库接入云芳邻后的处理要求

#### ① 保留现有能力

本期 MVP 保留原系统现有的视频上传、分类、搜索、分页和删除能力。【视频库】仍然使用原系统现有视频接口和存储配置，避免重做视频上传和转码能力。

#### ② 需要处理的点

| 事项 | 处理要求 | 原因 |
| --- | --- | --- |
| [添加分类] | 沿用原系统现有弹窗，保留「所属上级」「分类名称」 | 视频分类用于后台整理素材，不影响学员端课程分类 |
| [上传视频] | 保证视频存储配置可用 | 课时学习依赖视频资源，MVP 必须可用 |
| [修改视频分类] | 沿用原系统能力 | 分类只影响后台查找，不改变学员学习流程 |
| [删除视频] | MVP 阶段保留原能力；正式上线前至少增加二次确认 | 当前是嵌入使用，不建议为视频库单独做重改 |
| [视频使用情况] | 暂不作为 MVP 必做；后续如要二开，可增加“已绑定课时”提示 | 现有列表重点展示视频基础信息，不展示被哪些课时使用 |

#### ③ 验收重点

- 在【视频库】上传视频后，列表能看到视频名称、时长、大小和上传时间。
- 点击「添加分类」后，可以选择上级分类并填写分类名称，保存后分类出现在左侧分类树。
- 在课时管理中能选择或绑定视频库里的视频。
- 删除视频前有明确确认提示，避免运营人员误删课时素材。

#### ③ 验收重点

- 学员详情能展示所属区县、所属街道、所属社区和用户角色。
- 页面只保留学习相关内容。
- 点击课程行的「学习进度」打开弹窗，不跳转到新页面。

## PC端 · 数据菜单-学员数据

<!-- prototype: src="prototype/pc-stats-member-source.html" width="1920" height="1080" scale="0.55" title="PC端原型 · 数据菜单-学员数据" -->

### 1. 保留 · 学员数据用于查看学习和新增情况

#### ① 这个页面做什么

【学员数据】用于查看教培系统内的学员学习情况和新增情况。

这里的「新注册学员」指云芳邻用户第一次进入教培系统，并生成教培系统学员映射。它不等同于云芳邻全量新用户。

区县、街道、社区统计是接入云芳邻后新增的统计能力，用来让区、街道和社区看到自己范围内的学习推进情况。原系统只提供基础学员数据，本期需要在接入时补上组织归属筛选和排名。

#### ② 页面保留哪些能力

| 区域 | 保留能力 | 说明 |
| --- | --- | --- |
| [核心指标] | 今日学习学员、今日新注册学员、总学员数 | 沿用原系统现有学员统计主卡片 |
| [时间范围] | 今日、近 7 天、近 30 天 | 作为页面全局筛选，影响顶部指标和下方统计 |
| [街道统计] | 按区县筛选，展示各街道学习学员、新注册学员、总学员数 | 用于街道层面查看学习推进情况 |
| [街道排序] | 支持点击今日学习学员、新注册学员、总学员数排序 | 方便运营快速找到学习推进较好或较弱的街道 |
| [社区统计] | 先选择区县和街道，再展示该街道下各社区学习学员、新注册学员、总学员数 | 社区统计只看当前街道下的社区，不做全区混排 |
| [社区排序] | 支持点击今日学习学员、新注册学员、总学员数排序 | 方便街道或社区负责人查看社区排名 |

#### ③ 本期去掉哪些能力

| 去掉内容 | 原因 |
| --- | --- |
| 付费学员榜 | 本期不做课程售卖 |
| 支付订单数、支付总金额 | 本期不做支付统计 |
| 新老学员成交趋势 | 成交属于付费购买场景 |
| 学习完成率 | 不作为本页新增统计项，原系统学员数据页没有该指标 |

### 2. 调整 · 学员数据接入云芳邻后的处理要求

| 事项 | 处理要求 | 原因 |
| --- | --- | --- |
| [学员来源] | 只统计已进入教培系统并生成教培系统学员映射的云芳邻用户 | 不把未进入教培的云芳邻用户算进教培学员 |
| [今日学习学员] | 按课时观看记录统计当天有学习行为的学员去重人数 | 和播放器上报的学习记录保持一致 |
| [今日新注册学员] | 按学员映射创建时间统计 | 反映当天有多少云芳邻用户首次进入教培 |
| [总学员数] | 按当前组织范围内的学员映射总数统计 | 反映教培系统实际覆盖人数 |
| [组织归属] | 使用云芳邻用户所属区县、街道、社区 | 原系统只记录注册地区，不能满足社区/街道统计 |
| [街道统计表] | 卡片内滚动展示当前区县下全部街道 | 避免页面过长，同时保留完整排名 |
| [社区统计表] | 卡片内滚动展示当前街道下全部社区 | 社区统计跟随街道筛选，不和其他街道混在一起 |
| [数据权限] | 后台人员只能查看自己权限范围内的区县、街道、社区 | 防止跨组织查看学员数据 |
| [统计刷新] | MVP 阶段按实时查询或短时间缓存处理 | 页面用于运营查看，不要求秒级大屏刷新 |

#### ③ 验收重点

- 数据菜单只保留【学员数据】一页。
- 页面不出现付费榜、支付订单数、支付总金额、新老成交趋势。
- 切换时间范围后，顶部三张指标卡和下方统计同步变化。
- 按街道统计支持区县筛选和排序，表格卡片内部可滚动查看完整街道排名。
- 按社区统计支持区县和街道筛选，并可滚动查看当前街道下完整社区排名。
- 「今日新注册学员」按“首次进入教培并生成学员映射”计算。
- 后台人员不能看到权限范围外的街道和社区数据。

## PC端 · 系统管理-管理人员

<!-- prototype: src="prototype/pc-system-admins-source.html" width="1920" height="1080" scale="0.55" title="PC端原型 · 系统管理-管理人员" -->

### 1. 保留 · 管理人员用于查看教培后台授权人员

#### ① 这个页面做什么

【管理人员】用于查看哪些云芳邻用户已经被授权进入教培 PC 后台。

这个页面不再承担独立账号创建职责。管理员来源是云芳邻授权：用户在云芳邻被授予「教培管理员」后，可以看到「教培系统」菜单；首次进入教培后台时，系统自动生成教培系统管理员映射。

#### ② 页面保留哪些能力

| 区域 | 保留能力 | 说明 |
| --- | --- | --- |
| [管理员列表] | 沿用原管理人员表格骨架 | 列表展示 ID、姓名、云芳邻用户 ID、登录时间、操作 |
| [云芳邻用户 ID] | 替换原「登录邮箱」位置 | 不再展示独立邮箱账号 |
| [教培系统管理员 ID] | 作为云芳邻用户 ID 下方辅助信息展示 | 方便和教培系统日志、操作记录对应 |
| [登录时间] | 展示最近一次进入教培后台时间 | 用于判断后台使用情况 |
| [查看] | 查看云芳邻用户 ID、教培系统管理员 ID、教培角色、首次进入时间、最近进入时间 | 用于排查映射和权限问题 |

#### ③ 本期去掉哪些能力

| 去掉内容 | 原因 |
| --- | --- |
| 新建管理员 | 管理员来源为云芳邻授权，不在教培后台手动创建 |
| 删除管理员 | 取消权限应回到云芳邻权限管理处理 |
| 修改登录邮箱和密码 | 不再使用独立登录入口 |
| 管理员角色配置 | 教培角色由云芳邻权限决定，不在教培后台单独维护 |
| 禁止登录 | 账号是否能进入教培后台由云芳邻权限决定，不在教培后台单独停用 |
| 系统配置 | 不作为页面菜单展示，视频存储、文件存储、播放器等配置走后台预置或运维配置 |
| 系统日志 | 不作为页面菜单展示，底层保留日志用于排查 |
| 功能模块 | 不开放原系统 模块开关 |

### 2. 调整 · 管理人员接入云芳邻后的处理要求

| 事项 | 处理要求 | 原因 |
| --- | --- | --- |
| [授权入口] | 在云芳邻用户/后台权限中授予「教培管理员」 | 统一从云芳邻控制谁能进入教培后台 |
| [菜单显示] | 用户被授予「教培管理员」后显示「教培系统」菜单 | 避免无权限用户看到入口 |
| [首次进入] | 用户第一次点击「教培系统」时，检查是否已有教培系统管理员映射；没有则自动创建 | 给研发一个明确的建号时机 |
| [再次进入] | 已有映射时直接换取教培后台登录态进入 | 保持无感进入 |
| [取消授权] | 云芳邻取消教培权限后，用户不再看到「教培系统」菜单；教培系统管理员映射保留为历史记录 | 保留历史操作追溯 |
| [日志记录] | 后台操作记录同时保存云芳邻用户 ID 和教培系统管理员 ID | 方便追溯真实操作者 |

#### ③ 验收重点

- 系统菜单下只出现【管理人员】。
- 页面不出现系统配置、系统日志、功能模块入口。
- 页面不提供新建管理员、删除管理员、修改登录邮箱和密码、禁止登录。
- 云芳邻用户获得「教培管理员」权限后，首次进入教培后台会自动生成教培系统管理员映射。
- 云芳邻取消权限后，该用户不能再进入教培后台。
