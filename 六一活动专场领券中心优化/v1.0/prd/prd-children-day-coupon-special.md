## 需求背景

### 业务背景

六一活动期间，社区希望把线下亲子活动、现场积分发放、商家优惠券兑换和到店领奖串起来。居民在现场参与活动后获得积分，马上进入【领券中心】查看「六一活动专场」优惠券，完成兑换或免费领取，再到对应商家或摊位出示券码核销领奖。

当前【领券中心】已有「全部」「免费领」「积分换」「新客专享」Tab 和优惠券列表，适合承接这次活动。为了赶上六一节点，本期不新增 PC 配置，不做正式活动专题后台，只在 C 端增加专场入口和名称搜索能力，运营通过优惠券名称约定完成专场归集。

同版本补充一个 PC 端商家管理小需求。当前后台人员在【商家管理】核对商家时，可以看到商家基础信息，但商家管理员、管理员手机号和最后登录时间没有形成完整的筛选、列表和详情展示闭环。运营要判断商家账号是否活跃时，需要再去【用户管理】查用户手机号和最后登录时间，核对效率低。本期只补充字段展示和查询能力，不改变商家配置和用户管理的原有数据。

同版本补充一个 PC 端云新声处理小需求。当前【云新声管理】中，事项的「问题解决部门」一直表示居民提交时匹配到的原始部门；历史上的「转接上级部门」并没有真正改变该字段，只是让白云区管理员能看到并处理这类记录。新场景下事项可能在街道、党委等同层级组织间多次移交，继续依赖「问题解决部门」或「问题状态」判断列表可见范围已经不够。本期新增「当前处理部门」和「可处理部门列表」，用「可处理部门列表」统一决定哪些组织能在列表看到该事项；「问题状态」回到展示和筛选用途。

### 本期不做

- 不做 PC 端活动专场配置。
- 不新增活动表、专题表或后台运营页面。
- 不改变商家创建优惠券流程。
- 不改变积分扣减、卡包、券码、商家核销现有规则。
- 不做活动结束后的自动下线规则，仍按优惠券自身可领时间和上下架状态判断。
- 不新增商家管理员配置流程，不改变【用户管理】现有手机号和最后登录时间来源。
- 不新增区级接管流程，不改变云新声居民上报入口。
- 不改变云新声「已解决」处理逻辑。

## C端 · 领券中心页

<!-- prototype: src="../prototype/coupon-center-children-day.html" width="375" height="812" scale="0.86" title="C端原型 · 六一活动专场领券中心" -->

### 1. 页面结构

#### 功能描述

【领券中心】服务居民现场领券和日常领券。用户从小程序入口进入后，顶部展示页面标题；标题下方展示分类 Tab；Tab 下方新增搜索栏；主体为优惠券卡片列表。

#### 页面区域

| 区域 | 展示内容 | 说明 |
| --- | --- | --- |
| 顶部导航 | 返回按钮、标题「领券中心」、微信胶囊 | 沿用小程序标准头部 |
| 顶部 Tab | 「全部」「六一活动专场」「免费领」「积分换」「新客专享」 | 支持横向展示，选中态蓝色下划线 |
| 搜索栏 | 默认提示文案「搜索优惠券名称」 | 本期新增，按券名查，无独立搜索按钮 |
| 列表区 | 券名、标签、领取时间、库存进度、商家、距离、领取按钮 | 沿用现有卡片信息 |
| 空状态 | 「暂无符合条件的优惠券」 | 搜索无结果或专场无券时展示 |
| 反馈层 | 积分兑换确认面板、Toast 轻提示 | 积分券兑换前展示确认面板；领取或兑换成功后展示结果提示 |

### 2. 顶部 Tab 调整

#### 展示规则

| Tab | 展示文案 | 点击后默认条件 | 说明 |
| --- | --- | --- | --- |
| 全部 | 「全部」 | 不限制领取方式，不限制券名 | 原有逻辑保留 |
| 六一活动专场 | 「六一活动专场」 | 券名包含「六一专场」 | 本期新增，放在第二位；可使用轻量活动字标样式突出入口 |
| 免费领 | 「免费领」 | 仅展示免费领取券 | 原有逻辑保留 |
| 积分换 | 「积分换」 | 仅展示需要积分兑换的券 | 原有逻辑保留 |
| 新客专享 | 「新客专享」 | 仅展示新客专享券 | 原有逻辑保留 |

#### 点击后果

- 点击任一 Tab 后，清空当前列表并从第一页重新加载。
- 切换到「六一活动专场」时，列表按专场规则重新加载。
- 从「六一活动专场」切回其他 Tab 时，列表按目标 Tab 重新加载。
- 用户点击搜索框清除按钮：只清空搜索框内的用户输入内容，不影响当前 Tab 自带的筛选条件；例如在「六一活动专场」下清空搜索后，仍保留「六一专场」固定筛选。

### 3. 新增 · 搜索栏

#### 功能描述

搜索栏用于按优惠券名称查找，放在 Tab 下方。用户输入关键词后，列表按输入内容刷新；键盘确认也可触发刷新。

#### 展示和交互

| 项目 | 规则 |
| --- | --- |
| 位置 | 顶部 Tab 下方，列表上方 |
| 默认文案 | 「搜索优惠券名称」 |
| 输入范围 | 优惠券名称，最长 20 个中文字符或等长字符 |
| 查询触发 | 输入关键词、键盘确认 |
| 清除触发 | 点击输入框右侧清除图标 |
| 清除结果 | 清空用户输入并刷新当前 Tab 列表 |

### 4. 六一专场筛选与组合搜索

#### 功能描述

「六一活动专场」是一个前端固定入口。用户切换到该 Tab 后，系统默认筛选优惠券名称包含「六一专场」的券；如果用户继续在搜索栏输入关键词，则在专场条件基础上继续按用户输入词筛选。

#### 查询规则

| 当前 Tab | 用户输入 | 实际匹配规则 | 可命中示例 |
| --- | --- | --- | --- |
| 全部 | 蛋糕 | 券名包含「蛋糕」 | 「儿童蛋糕体验券」 |
| 六一活动专场 | 空 | 券名包含「六一专场」 | 「六一专场-亲子烘焙券」 |
| 六一活动专场 | 蛋糕 | 券名同时包含「六一专场」和「蛋糕」 | 「六一专场-儿童蛋糕体验券」 |
| 六一活动专场 | 篮球 | 券名同时包含「六一专场」和「篮球」 | 「六一专场-篮球体验课一节」 |

#### 规则说明

- 固定词「六一专场」只在「六一活动专场」Tab 生效。
- 匹配字段为优惠券名称。
- 切换 Tab 不改变上架、可领时间、库存、领取上限、积分、新客限制等原有规则。
- 实现时不能把两个词拼成一整个搜索词。比如「六一专场 蛋糕」作为整段模糊匹配，可能查不到「六一专场-儿童蛋糕体验券」。后端需要按多个名称条件分别判断，并要求全部命中。

### 5. 积分兑换确认提示

#### 功能描述

现有积分券点击「兑换」后会展示积分兑换确认面板。本期只在该面板中新增红色提示文案，提醒用户积分兑换后不可退回，优惠券过期未使用将作废；面板原有券信息、积分信息和操作按钮保持不变。

#### 展示规则

| 新增内容 | 展示文案 | 展示要求 |
| --- | --- | --- |
| 红色提示语 | 「温馨提示：积分兑换后不可退回，优惠券过期未使用将自动作废，请确认可使用后再兑换。」 | 在现有积分兑换确认面板内展示，使用红色字体，放在确认按钮上方或积分信息下方 |

#### 点击后果

- 原有「取消」「确认兑换」点击逻辑保持不变。
- 本期只新增红色提示语，不改变积分扣减、券码、卡包、过期作废和商家核销规则。

## PC端 · 商家管理

<!-- prototype: src="../prototype/merchant-management-admin-fields.html" width="1920" height="1080" scale="0.48" title="PC端原型 · 商家管理字段补充" -->

### 1. 列表筛选项补充

#### 功能描述

【商家管理】列表筛选区新增「商家管理员」「商家管理员手机号」「最后登录时间」三个筛选项。后台人员可在原有商家名称、商家类型、联系人、关联社区条件基础上，继续按商家管理员账号信息查找商家。

「商家管理员」和「商家管理员手机号」是两个独立筛选条件，不能继续沿用「姓名（手机号）」拼接字段做搜索。

#### 筛选规则

| 筛选项 | 控件 | 默认值 | 查询规则 | 重置结果 |
| --- | --- | --- | --- | --- |
| 商家管理员 | 输入框 | 空 | 按商家配置中的「商家管理员」姓名匹配 | 清空 |
| 商家管理员手机号 | 输入框 | 空 | 按该管理员在【用户管理】中的手机号匹配 | 清空 |
| 最后登录时间 | 时间范围 | 空 | 按该管理员在【用户管理】中的最后登录时间范围筛选 | 清空开始时间和结束时间 |

#### 点击后果

- 点击「查询」后，列表从第一页重新加载，原有筛选项和新增筛选项组合生效。
- 点击「重置」后，清空包含新增三项在内的全部筛选条件，列表按默认条件重新加载。
- 未配置商家管理员的商家，在未选择管理员相关筛选条件时正常展示；一旦使用管理员姓名、管理员手机号或最后登录时间筛选，则不命中这些空值商家。

### 2. 列表字段补充

#### 功能描述

当前【商家管理】列表中，「商家管理员」字段为「姓名（手机号）」组合展示。本期明确调整为「商家管理员」「商家管理员手机号」「最后登录时间」三列，方便后台人员直接查看管理员身份、手机号和最近登录情况。

列表中「商家管理员」只展示姓名，「商家管理员手机号」只展示手机号，两个字段分别成列，不做拼接展示。

#### 表格列规则

| 新增字段 | 展示内容 | 字段来源 | 无值处理 |
| --- | --- | --- | --- |
| 商家管理员 | 商家配置中选择的管理员姓名 | 商家配置里的「商家管理员」 | 单元格留空 |
| 商家管理员手机号 | 该管理员在【用户管理】中的手机号 | 用户管理 | 单元格留空 |
| 最后登录时间 | 该管理员在【用户管理】中的最后登录时间 | 用户管理 | 单元格留空 |

#### 展示要求

- 新增三列展示在商家基础信息列之后、状态类字段之前。
- 当前「商家管理员」列展示为「姓名（手机号）」，本期调整为姓名和手机号两列分别展示。
- 列表为空值时保持空白，不显示「-」或其他占位符。
- 原有商家名称、商家类型、联系人、关联社区、营业状态、商家状态、更新时间、操作列保持不变。
- 行内「商家详情」「修改」「更多」操作保持原有逻辑。

### 3. 详情弹窗补充

#### 功能描述

商家详情弹窗的「商家信息」Tab 中，原「商家管理员」字段后增加展示「最后登录时间」。后台人员进入商家详情时，可以直接判断该管理员账号最近是否登录过。

#### 展示规则

| 页面位置 | 展示方式 | 无值处理 |
| --- | --- | --- |

## PC端 · 云新声管理

<!-- prototype: src="../prototype/yunxinsheng-transfer-handler.html" width="1920" height="1080" scale="0.48" title="PC端原型 · 云新声移交其他部门处理" -->

### 1. 处理范围调整

#### 功能描述

云新声事项原本只在「问题解决部门」和白云区之间流转。新业务下，事项可能从街道移交给街道，也可能从街道移交给党委，并且后续还可能再次移交。

本期保留「问题解决部门」作为居民提交时的原始归属，同时新增「当前处理部门」和「可处理部门列表」，用于支持事项在多个组织之间持续处理。

#### 业务规则

| 项目 | 含义 | 规则 |
| --- | --- | --- |
| 问题解决部门 | 继续表示居民提交时匹配到的原始部门 | 不随移交变化 |
| 当前处理部门 | 表示当前主要应处理该事项的组织 | 新提交时为问题解决部门；移交后更新为目标处理组织；多部门先后操作时，以最后一条操作记录指派的部门为准 |
| 可处理部门列表 | 表示哪些组织可以在列表看到该事项 | 新提交时包含问题解决部门；每次移交时追加目标处理组织 |
| 问题状态 | 表示事项当前处理结果 | 用于展示和筛选，不再决定组织是否可见 |

#### 历史数据处理

| 数据类型 | 补齐规则 |
| --- | --- |
| 新提交事项 | 当前处理部门写为问题解决部门，可处理部门列表写入问题解决部门 |
| 待部门解决 | 当前处理部门写为问题解决部门，可处理部门列表写入问题解决部门 |
| 转接上级部门 | 当前处理部门写为白云区，可处理部门列表写入问题解决部门和白云区 |
| 已解决 | 当前处理部门优先取最后一次处理组织；没有处理记录时取问题解决部门；可处理部门列表至少写入问题解决部门 |

### 2. 列表与详情展示

#### 功能描述

后台人员进入【云新声管理】时，只能看到自己所属组织可处理的事项。系统先判断当前组织是否在事项的「可处理部门列表」中，再按页面筛选条件展示结果。

#### 展示规则

| 项目 | 规则 |
| --- | --- |
| 列表可见 | 当前登录组织在「可处理部门列表」中，则可看到该记录 |
| 问题状态筛选 | 保留现有状态筛选，并新增「移交其他部门处理」 |
| 问题解决部门 | 继续展示居民提交时的原始归属部门 |
| 当前处理部门 | 列表和详情中展示，方便后台判断当前主要承接组织 |
| 可处理部门列表 | 原型中展示，用于核对哪些组织能看到该事项 |
| 办理入口 | 记录在列表中可见，且状态不是「已解决」时，沿用现有「办理」入口 |

### 3. 办理移交与处理记录

#### 功能描述

管理员在办理弹窗中可以选择「已解决」或「移交其他部门处理」。选择「移交其他部门处理」时，需要选择一个目标处理组织，并填写移交原因。

提交后，事项进入新的处理状态；目标组织成为当前处理部门，也会加入可处理部门列表。后续多个组织继续处理时，每次操作都会在「处理情况」中留底。

#### 办理规则

| 项目 | 规则 |
| --- | --- |
| 办理弹窗状态 | 「问题状态」选项展示「已解决」「移交其他部门处理」 |
| 目标处理组织 | 下拉单选，展示街道层级可处理组织 |
| 提交校验 | 选择「移交其他部门处理」时，必须选择目标处理组织 |
| 提交后状态 | 问题状态更新为「移交其他部门处理」 |
| 提交后部门 | 当前处理部门更新为目标处理组织；目标组织追加进可处理部门列表 |
| 处理记录 | 每次办理都追加一条处理记录，记录问题状态、原因说明、操作人、操作人所属组织和操作时间 |
| 多次处理 | 多个组织先后操作时，处理记录逐条展示；当前处理部门以最后一条操作记录指派的部门为准 |

## 接口设计

### 1. 领券中心列表查询逻辑

#### 功能描述

用于【领券中心】列表加载。本期只在原有列表查询基础上增加“按优惠券名称搜索”的业务能力，不改变原有券状态、库存、排序、距离、可领时间和用户领取状态判断。

#### 后端处理规则

| 场景 | 查询规则 |
| --- | --- |
| 普通 Tab，无搜索 | 按原 Tab 条件查询 |
| 普通 Tab，有搜索 | 原 Tab 条件 + 优惠券名称包含用户搜索词 |
| 六一活动专场，无搜索 | 优惠券名称包含「六一专场」 |
| 六一活动专场，有搜索 | 优惠券名称包含「六一专场」且包含用户搜索词 |

#### 组合条件说明

六一活动专场的固定词和用户输入词要作为两个独立名称条件处理，并且要求全部命中。实现时不要把两个词拼接成一个长词做模糊匹配，避免「六一专场 蛋糕」查不到「六一专场-儿童蛋糕体验券」。
