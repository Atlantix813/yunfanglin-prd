## 需求背景

现有积分体系存在以下问题：

- 积分获取渠道单一（仅注册、邀新、参与活动），用户缺乏持续获取动力
- 积分无有效期，长期沉淀导致贬值风险
- 缺少统一的任务引擎，行为触发与积分发放逻辑散落在各业务模块

---

| 目标类型 | 描述 | 衡量指标 |
|---|---|---|
| **核心业务** | 建立积分月度桶 + FIFO扣减 + 12个月有效期 | 积分过期回收率 |

## C端 · 任务中心页

<!-- prototype: src="prototype/task-center.html" width="375" height="812" scale="0.85" title="C端原型 · 任务中心" -->

#### 功能描述

### 1. 页面结构

- 顶部：积分余额卡（蓝色渐变卡片）
  - 左上：「我的积分」标签 + 积分总余额（大字号）
  - 底部右侧：「积分明细」文字入口（半透明白色），点击跳转【我的账户】页；「积分兑换」胶囊按钮（白底蓝字，视觉突出），点击跳转【积分兑换】页，引导用户消费积分
  - 数据来源：接口返回用户当前可用积分余额（所有未过期积分之和）
- 中部：Tab栏（全部 / 待完成 / 已完成）
  - 选中态：加粗 + 蓝色下划线
  - 向下滚动时Tab栏吸顶固定，保持可切换状态
  - 切换时筛选下方任务列表，无需重新请求接口
- 下部：任务卡片列表，按状态排序展示

### 2. 任务卡片结构

每张卡片由三部分组成：

- **左侧图标**：44x44px 圆角方形色块，不同行为类型对应不同颜色，已完成状态统一置灰
- **中间内容区**：
  - 第一行：任务名称（16px加粗）+ 积分徽章（橙底圆角标签，星星图标 + "N积分"），来源于后台配置的[任务名称]和[积分值]字段
  - 第二行：任务说明文字（12px灰色），来源于后台配置的[任务描述]字段
- **右侧操作区**：
  - 操作按钮（「领取奖励」/「去完成」/「已达上限」/「已完成」）
  - 周期进度文字（如"每日 1/3"、"每月 2/60"、"一次性 1/1"），来源于[上限规则]和当前已完成次数

### 3. 任务状态与交互流程

任务卡片有4种状态，完整流转如下：

**「去完成」（蓝色按钮）**
- 点击后跳转至该任务配置的[去完成跳转目标]（详见预置任务清单中"去完成跳转目标"列）
- 用户完成一次行为即触发系统创建任务明细（关联流程2：任务触发流程）
- 若该任务开启了「自动发放」：积分直接到账，周期进度 +1；若未达上限则可继续完成下一次，若已达上限则变为「已达上限」
- 若未开启「自动发放」：状态切换为「领取奖励」，等待用户手动领取

**「领取奖励」（橙色按钮，仅未开启自动发放的任务）**
- 完成行为后按钮自动切换为「领取奖励」，卡片置顶
- 点击后弹出 Toast 提示："任务完成，获得 X 积分"（X为该任务配置的积分值）
- 积分写入用户账户（走流程6：用户领取任务奖励）
- 领取后周期进度 +1（如 1/3 变为 2/3），状态回到「去完成」
- 若周期进度已满（如 3/3），则变为「已达上限」

**「已达上限」（灰色按钮）**
- 当前周期（日/周/月）的完成次数已达上限
- 卡片整体置灰（opacity:0.75），沉底排列
- 点击弹出 Toast："任务完成数已达上限"
- 下一周期开始时自动重置为「去完成」（由流程3每日定时任务处理）

**「已完成」（灰色按钮）**
- 仅用于一次性任务（上限规则为"仅1次"），完成后永久展示此状态
- 卡片置灰沉底，点击弹出 Toast："任务完成数已达上限"

### 4. 状态判定与样式映射

前端展示状态由任务明细记录 + 任务配置 + 周期进度综合判定：

| 按钮显示 | 前端判定逻辑 | 后端数据条件 | 按钮样式 | 卡片表现 |
|---|---|---|---|---|
|「领取奖励」| 该任务存在一条待领取的明细记录 | 任务明细状态 = [待领取奖励] | 橙色填充 + 投影 | 正常显示，置顶 |
|「去完成」| 该任务在当前周期内无活跃明细，用户尚未参与 | 无任务明细记录 | 蓝色填充 + 投影 | 正常显示 |
|「已达上限」| 周期内上限为N次（N>1）的任务，当前周期已领取次数达到上限 | 任务明细状态 = [已领取奖励]，周期内记录数 >= 上限次数（N>1） | 灰色填充，无投影 | 整卡置灰，沉底 |
|「已完成」| 周期内上限为仅1次的任务，已领取完毕 | 任务明细状态 = [已领取奖励]，上限次数 = 1 | 灰色填充，无投影 | 整卡置灰，图标灰色，沉底 |

说明：
- 「已达上限」和「已完成」在后端都是[已领取奖励]，前端根据上限次数（仅1次 vs N次）区分展示文案
- 自动发放的任务不会停留在[待领取奖励]状态，行为触发后直接变为[已领取奖励]

### 5. 排序规则

列表按以下优先级从上到下排列：

1. 待领取奖励（最前，引导用户立即领取）
2. 待完成（尚未完成行为的任务）
3. 已达上限 / 已完成（最后，置灰）

同级内按任务配置的序号排列（即预置任务清单中的序号）。

### 6. Tab筛选逻辑

| Tab | 展示范围 |
|---|---|
| 全部 | 所有状态的任务 |
| 待完成 | 待领取奖励 + 有任务配置但用户还未完成的任务（可操作） |
| 已完成 | 已达上限（周期内次数用完）+ 已完成（一次性任务已完成），本周期已无法操作的任务 |

### 7. 数据来源与接口

- 任务列表：查询任务配置中状态为"上架"的记录，关联当前用户的任务明细获取进度
- 积分余额：用户所有未过期积分之和
- 周期进度：当前周期内该用户已完成该任务的次数

### 8. 异常与边界

| 场景 | 处理方式 |
|---|---|
| 无任何任务 | 空状态插画 + 文案"暂无可用任务" |
| 网络异常 | Toast "网络异常，请稍后重试"，保留本地缓存列表 |
| 领取时积分写入失败 | Toast "领取失败，请重试"，状态不变 |
| 任务被后台下架 | 下次刷新时从列表消失，已领取的积分不受影响；已产生的待领取明细不受影响，用户可在有效期内继续领取（通过消息通知等入口），超期未领则自动过期 |
| 跳转链接为空 | 「去完成」按钮仍可点击，但不跳转，仅依赖行为触发自动完成 |

- **标题格式**：`任务奖励-{任务名称}`（如"任务奖励-发布邻里帖子"）
- **金额**：正数，显示为 `+N`
- **时间**：领取奖励的时间
- 同样纳入月度收支汇总的"收入"统计
- **流水双方**：积分支出方为平台，收入方为用户；后台需对应生成一条平台支出流水和一条用户收入流水
- 过期清零记录标题为"积分过期清零"，金额为负数，附带说明"月度积分到期自动清零"

## PC端 · 任务配置管理

<!-- prototype: src="prototype/task-config.html" width="1920" height="1080" scale="0.35" title="PC端原型 · 任务配置列表" -->

#### 设计原则

任务触发逻辑由开发预埋（每条任务对应一个系统内置的行为事件），运营后台仅负责调整参数和管理开关。系统初始化时写入全部预置任务，不支持新增或删除。

#### 功能描述

### 1. 列表页

- **筛选区**：支持按[任务名称]（模糊搜索，同时匹配行为类型标签和任务名称）、[上架状态]（下拉：全部/已上架/已下架）组合筛选
- **表格列**：

| 列名 | 说明 |
|---|---|
| [序号] | 自增序号 |
| [行为类型] | 语义标签（如"浏览信息公开"），系统预置不可修改 |
| [任务名称] | 运营可编辑的C端展示文案 |
| [任务描述] | 运营可编辑的C端展示描述，溢出省略 |
| [积分值] | 数值显示 |
| [上限规则] | 显示为"每日 3次"格式，无上限则显示"无上限" |
| [自动发放] | 是/否标签样式 |
| [上架状态] | 标签样式：已上架(绿)、已下架(灰) |
| [操作] |「编辑」「上架/下架」 |

- **行内操作**：
  - 「编辑」：打开编辑任务配置弹窗
  - 「上架/下架」：弹出二次确认弹窗，确认后执行
    - 下架确认：标题"确认下架此任务？"，警告图标（橙色三角），说明文案"下架后，C端用户将无法在页面中看到该任务。但已被用户接取的任务明细，在有效期内仍可正常完成和领取。"，按钮「取消」+「确定下架」
    - 上架确认：标题"确认上架此任务？"，信息图标（蓝色圆圈），说明文案"上架后任务将对C端用户可见，用户可在任务中心看到并完成该任务。"，按钮「取消」+「确定上架」
- **无分页**：共18条预置任务，单页展示全部

### 2. 编辑弹窗

弹窗标题固定为"编辑任务配置"，分为只读区和可编辑区：

**只读信息区**：

| 字段 | 说明 |
|---|---|
| 行为类型 | 语义化标签（如"浏览信息公开"、"邻里圈发帖"），系统预置不可修改，结合功能模块和行为自动生成 |

**可编辑参数区**：

| 字段 | 控件类型 | 规则说明 |
|---|---|---|
| [任务名称] | 文本输入 | 必填，最多20字，C端任务卡片主标题 |
| [任务描述] | 文本域 | 选填，最多100字，C端任务卡片描述行 |
| [积分值] | 数字输入 + "积分"后缀 | 必填，根据任务类型动态提示建议范围：常规任务 1 ~ 10；超出范围红色警告但允许保存 |
| [上限规则] | 勾选"无上限" 或 周期下拉 + 次数输入 | 周期选项：每日/每周/每月/仅；勾选无上限时禁用周期和次数输入 |
| [有效期] | 单选组（永久/限时） | 选择限时时展开起止时间选择器 |
| [自动发放] | 开关 | 开启后完成任务自动发放积分，无需用户手动领取；默认关闭 |
| [是否上架] | 开关 | 上架后任务对C端用户可见 |

- 点击「保存」更新参数，即时生效，显示保存成功提示

- [任务名称]
- [任务描述]
- [积分值]
- [上限规则]
- [自动发放]状态

快照用途：

- 积分发放金额以快照中的[积分值]为准，不随后续配置调整变化

## 订阅消息

<!-- prototype: src="prototype/notification-settings.html" width="375" height="812" scale="0.85" title="C端原型 · 消息订阅设置" -->

#### 功能描述

复用云芳邻已有的微信长期订阅消息能力（总开关 + 业务子开关 + 统一长期订阅模板），本版本新增「积分提醒」业务子开关，覆盖 2 类通知场景：任务完成提醒、积分过期提醒。

### 1. 新增子开关

- **位置**：【消息订阅设置】页，业务子开关分类下
- **名称**：积分提醒（2 项）
- **默认状态**：用户首次进入时默认开启（与现有子开关默认策略保持一致）
- **关闭后**：不再向该用户推送积分相关通知，已开启的微信总开关授权不受影响
- **子项**：【积分提醒】任务完成提醒、【积分提醒】积分过期提醒

### 2. 任务完成提醒

**触发事件**

用户在业务页面完成行为，系统命中某条任务的触发条件并创建任务明细。仅当该任务 [自动发放] = 关闭、且创建出的任务明细状态 = 「待领取奖励」时触发推送。[自动发放] = 开启的任务积分直接到账，不发送本消息。

**模板字段填充**

| 字段 | 内容 |
|---|---|
| 社区通知 | 任务完成提醒 |
| 通知内容 | 您完成了「{任务名称}」，{积分值} 积分待领取 |
| 通知时间 | 取任务明细的 [完成时间] 字段，格式 YYYY-MM-DD HH:mm |
| 提示说明 | 点击进入任务中心立即领取 |

> 模板字段中 `{任务名称}`、`{积分值}` 取自该条明细的快照数据，不随后续任务配置变化。

**落地页**

用户点击订阅消息卡片 → 跳转【任务中心】页面，由用户点击对应卡片的「领取奖励」按钮完成领取。

**频次控制**

- 同一用户一小时内最多收到 1 条任务完成提醒
- 以用户最近一次收到该类推送的时间为起点，窗口内新产生的「待领取奖励」明细静默跳过，不排队不补发
- 窗口内第一条命中的任务决定推送文案

### 3. 积分过期提醒

**触发事件**

系统每日定时检查（可复用流程5积分桶过期清零的定时任务），当用户存在 30 天内即将过期的积分时触发。

**模板字段填充**

| 字段 | 内容 |
|---|---|
| 社区通知 | 积分过期提醒 |
| 通知内容 | 您有 {即将过期积分数} 积分将于 {最近过期日期时间} 过期，请尽快使用 |
| 通知时间 | 提醒发送时间，格式 YYYY-MM-DD HH:mm |
| 提示说明 | 点击查看积分明细 |

> `{即将过期积分数}` 和 `{最近过期日期时间}` 的数据来源与【我的账户页 - 1. 过期积分提示条】一致，查询用户未过期积分中过期时间在未来 30 天内的记录汇总。

**落地页**

用户点击订阅消息卡片 → 跳转【我的账户】页面，查看积分余额并消费。

**频次控制**

- 同一批过期积分推送 2 次：过期前 30 天推送第 1 次，过期前 3 天推送最后 1 次
- 已推送过的批次不重复推送（以积分桶月份 + 用户为去重维度）
