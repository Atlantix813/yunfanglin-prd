## 🏗️ 设计架构 (Architecture)
*项目底层模型快照*

- **核心数据模型**（详见 `积分任务系统_底表结构设计.md`）：
	- `task_config` —— 任务配置
	- `task_detail` —— 任务明细（用户对任务的执行实例，包含 behavior_type 冗余列和 task_name / task_desc / points_value / limit_rule / auto_award 冻结列）
	- `task_config_change_log` —— 配置变更审计
	- `user_points_bucket` —— 积分月度桶（12 个月有效期）
	- `bucket_deduction_log` —— 跨桶 FIFO 扣减日志
	- `committee_points_grant_log` —— 居委手动发放记录

- **流程图 / 原型**（Figma `mwjTAI3cUEI582mTVwMAwm`）：
	- 流程1：任务中心展示 —— 左路配置驱动可接取任务 / 右路活跃明细（status=1(待领取奖励)）直接用冻结字段展示
	- 流程2：任务触发流程 —— 先查活跃明细再查配置；status=1(待领取) 忽略 / 未命中走创建 → 判断auto_award → 自动发放或待手动领取
	- 流程3：每日定时任务 —— 周期重置 + 待领取自动领取

- **与现有系统的依赖关系**：
	- **积分流水表**：现有的"双方账簿"结构（收入方 + 支出方 + 双方结余 + 已退还）保留复用，不重建
	- **组织账户**（社区居委/平台池）：保持单一余额模型，**不参与月度分桶**
	- **个人账户余额**：从"单余额"改为"用户所有桶的 SUM(remaining)"

## 🔑 关键决策 (Decision Log)
*记录设计过程中定型的关键决策与理由*

* **D-01：统一平台出资，去掉scope和community_id字段**
	* 决策：统一平台出资，去掉scope和community_id字段。4月20日会议已确认任务统一由平台配置、平台出资，居委不再有任务配置权限
	* 理由：4月20日会议确认业务简化方向
	* 影响：task_config 去掉 scope、community_id 列；所有任务统一从平台池出资

* **D-02：去掉 biz_id**
	* 决策：任务配置不关联具体业务实体（活动ID / 问卷ID 等）
	* 理由：业务上不需要"参加某场特定活动 vs 参加任何活动"的粒度
	* 影响：点查配置从 Q1-Q4 简化为 Q1-Q2；UNIQUE 改为 `(behavior_type, scope, community_id)`；流程无双发/同键覆盖问题

* **D-03：同 behavior_type 下不会跨出资方冲突**
	* 决策：A/B 类行为不存在交集，同一 behavior 只能由一方配置
	* 理由：业务职责分明
	* 影响：候选配置天然最多命中 1 条，无需同键覆盖逻辑

* **D-04：task_detail 去掉 snapshot JSON，改独立冻结列**
	* 决策：用 `task_name`、`task_desc`、`points_value`、`limit_rule`、`auto_award` 五个独立列冻结配置值
	* 理由：JSON 查询性能差、维护成本高；只有"可变字段"才需要冻结保护（behavior_type、funder 都是不可变字段，无需冻结）
	* 影响：明细表字段更清爽，查询不用 JSON 函数

* **D-05：task_detail 冗余 behavior_type 一等列**
	* 决策：task_detail 直接有 behavior_type 列，不依赖 JOIN task_config 获取
	* 理由：行为触发是最高频查询路径；behavior_type 不可变无一致性风险；task_config 被删除时明细仍能独立工作
	* 影响：索引 `(user_id, behavior_type, status)` 支撑流程2 短路查询

* **D-06：明细状态命名**
	* 决策：**1=待领取奖励**，**2=已领取奖励**。去掉进行中（无多步进度概念）和已过期（到期自动领取，不存在过期）
	* 理由：简化为2态，匹配单次触发即完成的业务模型
	* 影响：全部流程图、PRD、代码统一口径

* **D-07：status=2 待领取时忽略新行为**
	* 决策：用户有待领取奖励的明细时，同行为的新触发直接忽略，不开新一轮
	* 理由：强制"完成→领取→才能开下一轮"的顺序性，避免用户囤积未领奖励 + 同时开多轮
	* 影响：流程2 中该分支直接到结束节点，不走周期上限校验

* **D-08：周期上限按当前周期内明细总记录数计算**
	* 决策：`count(明细 WHERE created_at in 当前周期) ≥ limit_count` 即算已达上限
	* 理由：同周期内明细不会出现 status=4（明细 expire_time=周期末尾，过期只在周期切换时发生），status=1/2/3 都占槽位；简单无歧义
	* 影响：周期起始/结束时间计算细节交给开发实现

* **D-09：流程1 活跃明细独立展示，不查配置**
	* 决策：活跃明细（status=1 待领取奖励）用明细上的冻结字段直接渲染
	* 理由：任务下架/修改/删除场景下，已接取用户进度仍能正常完成
	* 影响：流程1 左右双路分工——配置路生成可接取任务，明细路独立走完

* **D-10：流程2 先查活跃明细再查配置**
	* 决策：行为触发时第一步查用户活跃明细，未命中才走配置点查
	* 理由：业务正确性（任务下架后已接取仍能推进）+ 高频路径性能
	* 影响：流程2 第①步是查明细，配置相关判断全在未命中分支

* **D-11：同步双发场景已消解**
	* 决策：不需要同键覆盖 / 双发排他逻辑
	* 理由：D-02 和 D-03 联合作用后，已不存在同行为多条配置并存的场景
	* 影响：流程1/2 大幅简化

* **D-12：任务领取不独立写日志**
	* 决策：积分流水表承担全量积分变动审计，不为流程3 / 手动领取单独建日志表
	* 理由：流水表结构已足够承载（事项、变动类型、收支双方、结余等）；冗余日志增加维护成本
	* 影响：流程3 移除"记录领取日志"节点

* **D-13：组织账户不分桶**
	* 决策：积分月度桶仅用于个人用户，组织账户（社区居委/平台池）保持单一余额
	* 理由：组织账户的积分来源（平台划拨 / 用户兑换回收）和消耗（发放给用户）已有独立管理机制（运营方案中的"每月划拨+用进废退递减"）
	* 影响：桶相关字段、FIFO 扣减、过期回收都只作用于 user_points_bucket

* **D-14：积分桶用独立 `user_points_bucket` 表，不扩展积分明细**
	* 决策：新开独立桶表，不在现有积分流水表上加 remaining 等字段
	* 理由：
		- 审计不可变原则：流水是历史快照，不应该被反复 UPDATE
		- 性能：桶是聚合小单位（每用户每月 1 条），余额 SUM 极快；扩展流水则要扫全部收入记录
		- 过期清零风险隔离：操作桶表影响小，批量 UPDATE 流水风险大
		- 粒度对齐：业务定义是"按月归桶"，桶表结构直接匹配
	* 影响：新建 user_points_bucket + bucket_deduction_log 两张表；流水表保持不变

* **D-15：积分桶按月划分**
	* 决策：桶的粒度 = (用户, 月份)，一个用户每月最多 1 条桶记录
	* 理由：**业务定义"12 个月有效期 + 每月集中过期"决定了月是最小有效粒度**，其他都是衍生收益
		- 同月入账的积分过期时间本来就相同，按笔拆桶是冗余
		- 过期清零可以每月 1 日批量跑一次即可，简单可靠
		- 数据规模：10 万活跃用户一年 120 万条（按笔会到千万级）
		- FIFO 扣减时扫的桶数少（用户最多 12 个活跃桶）
	* 影响：`expire_time = bucket_month 首日 + 12 个月`；定时任务每月 1 日凌晨触发

* **D-16：流水与桶的关联用独立关联表，不用 JSON / 逗号分隔**
	* 决策：新建 `flow_bucket_detail` 表（原名 `bucket_deduction_log`），记录每条流水与桶的多对多关系
	* 理由：
		- JSON / 逗号分隔字段无法走索引，按桶反查流水、聚合对账都低效
		- 关联表支持 FK 约束、数据完整性有数据库层保障
		- 跨桶消耗数据量可控（百万级/年），不需要优化性能到 JSON 存储
	* 影响：关联表采用"流水-桶"标准一对多建模，不是"日志表"

* **D-17：入账和扣减统一用 flow_bucket_detail 记录**
	* 决策：入账（amount 正）和扣减（amount 负）都写入 flow_bucket_detail，用 direction 字段区分 credit / debit
	* 理由：
		- 桶视角可完整追溯所有进出历史（`WHERE bucket_id=X ORDER BY time`）
		- 退款逻辑统一（反向再写一条），不需要区分"收入的退 vs 扣减的退"
		- 对账不变量统一：`桶 remaining = SUM(amount WHERE bucket_id=X)`
	* 影响：
		- 入账：1 条流水 + 1 条 detail（正值）+ 更新桶 total_earned & remaining
		- 消耗：1 条流水 + N 条 detail（负值）+ 逐桶扣减 remaining

* **D-18：关联表原名 bucket_deduction_log 重命名为 flow_bucket_detail**
	* 决策：表名从"扣减日志"语义改为"流水桶明细"，对齐 D-17 的统一记录定位
	* 理由：原名暗示"仅扣减"，与 D-17 冲突；新名中性，准确描述一对多关系的本质

* **D-19：桶过期时间基准改为"次月首日 + 12 个月"**
	* 决策：`expire_time = bucket_month 次月首日 00:00:00 + 12 个月`
	* 理由：
		- 原规则"bucket_month 首日 + 12 个月"下，月末获得的积分实际有效期不到 12 个月，与宣传口径"12 个月有效期"冲突
		- 新规则保证月内任何时间获得的积分都至少有完整 12 个月可用
		- 避免客服/客诉争议
	* 影响：
		- 用户活跃桶数从最多 12 个变为最多 13 个（略增，无影响）
		- 过期清零定时任务逻辑不变，仍每月 1 日凌晨跑

* **D-21：桶操作顺序——先写积分流水，再做桶同步**
	* 决策：业务流程中，积分流水的写入**先于**桶变动；桶服务是在流水已存在的基础上做"桶层面的同步"
	* 理由：
		- 现有积分流水业务已成熟运行，不改动业务侧代码，改动面最小
		- 桶服务拿到流水 ID 后才能写 flow_bucket_detail，顺序自然
		- 流水本身是审计主记录，优先落地保证核心审计能力不丢失
	* 影响 / 研发自理项：
		- 产品侧只定义业务流程顺序，不约束研发实现
		- 原子性保证方案（大事务扩展 / 桶失败补偿流水 / 最终一致 + 重试）由研发结合系统架构自行选型
		- 设计前提假设：桶操作失败率极低，即使需要补偿也是小概率

* **D-20：C2C 转账采用"过期时间继承"方案，防止洗过期**
	* 决策：居民之间转账积分时，接收方按"源桶归属月份"归入桶，新建桶时过期时间**继承自源桶**，不使用转账发生月
	* 理由：
		- 若接收方进当月桶，积分过期时间被刷新，左右倒手即可无限续命，"12 个月过期"规则完全失效
		- 继承源桶过期时间 → 积分所有权转移，但过期时间从获得那刻起就绑死，不受转账次数影响
		- 砍掉转账功能代价太大，其他补丁方案（临期禁转、永久桶）都不够严密
	* 影响：
		- 桶的 `bucket_month` 含义扩展：代表"归属月份"，不一定是"创建月份"
		- 接收方可能出现"自己没赚过该月份积分，但因转账入账有该月桶"的情况
		- 转账流程需专门处理"A 扣 N 条桶 → B 按源桶月份入 N 条桶"（流程4-1）
		- 组织账户 ↔ 用户的积分变动不涉及继承（组织账户不分桶），发放进用户当月桶即可

* **D-22：任务配置改为20条预置种子数据**
	* 决策：任务配置改为20条预置种子数据，系统初始化写入，不支持新增或删除。触发逻辑由开发预埋，运营后台仅管理参数和开关。
	* 理由：任务类型固定，触发逻辑需开发预埋，运营自行新增无意义
	* 影响：运营后台仅提供参数编辑和开关管理，不提供新增/删除入口

* **D-23：新增auto_award字段，支持自动发放模式**
	* 决策：新增auto_award字段，支持自动发放模式。开启后行为触发时直接创建已领取状态的明细并发放积分，无需用户手动领取。建议对被动/异步触发类任务（邀请好友、审核通过类）开启。
	* 理由：被动触发类任务用户无法主动感知完成，强制手动领取体验差
	* 影响：流程2 新增 auto_award 判断分支；task_detail 新增 auto_award 冻结列

* **D-24：去掉单次进度/进度条概念**
	* 决策：去掉单次进度/进度条概念。当前20条任务均为单次行为即触发，不需要多步动作累积。progress_required和current_progress/max_progress字段从底表移除。此设计作为封存方案，后续如有聚合型任务需求可启用。
	* 理由：当前所有任务均为单次触发，进度条字段无实际使用场景
	* 影响：task_config 移除 progress_required；task_detail 移除 current_progress、max_progress

	* 现状："居民设金额→居委扫码支付"
	* 目标："居委主动发起"
	* 具体交互流程待定

* **Q-06：旧积分流水"一条支出一条收入"双写模型是否改单写**
	* 现状（推测需后端确认）：每次交易写 2 条流水（收入/支出各一条）

	* 决策前提：需先和后端确认双写的历史原因和底层实现
